# UI Design：2D 像素风做菜 HUD 重构

本文档记录当前 UI 重构方向。该方案不替代 `docs/mvp.md`，也不扩展排行榜、NPC 自动品鉴等非当前任务内容。

## 1. 重构目标

当前首要问题是做菜界面过于像表单和按钮面板，缺少游戏感。UI 重构优先解决做菜页，不优先重做登录、模式选择、评价、保存和食单页面。

目标风格：

- 2D 像素风厨房。
- 中央是可观察的锅、灶台、食材和调料区域。
- 玩家在准备阶段通过简单交互推进流程。
- 玩家在炒菜阶段可以自由调整火候、油、水和调料量。
- 所有关键操作写入日志，作为后续本地评价和 AI 评价的批判依据。

## 2. 页面范围

第一版 UI 重构只重点改造做菜页。

保留现有基础流程：

`登录 -> 模式选择 -> 菜谱选择 -> 做菜 HUD -> 评价 -> 保存菜品 -> 食单`

其他页面只保持可用和基础一致性，不做深度美术重构。

## 3. 做菜 HUD 布局

做菜页建议拆为以下区域：

- 顶部状态栏：显示厨神名、声望、当前菜名。
- 中央厨房区：像素风灶台、锅、食材、调料、火焰或热度反馈。
- 准备交互区：食材准备、切菜、下锅等一次性交互。
- 炒菜控制区：火候、油量、水量、调料量等滑杆或等价控件。
- 操作反馈区：当前状态、当前提示、非法操作提示、出锅按钮。

中央厨房区是视觉重点，按钮和滑杆只作为操作工具，不应压过锅和食材的展示。

## 4. 交互设计

### 4.1 准备阶段

准备阶段采用单次交互，不做复杂小游戏。

第一版包含：

- 食材准备：点击或拖动食材到准备区。
- 切菜：点击案板或食材完成切菜。
- 下锅：点击或拖动处理后的食材到锅。

每个准备动作需要有明确视觉反馈，例如食材图标变形、进入锅中、状态文本更新。

### 4.2 炒菜阶段

炒菜阶段提供更高自由度。

第一版包含：

- 火候控制：滑杆调整当前火力。
- 油量控制：滑杆或加油按钮记录加入油量。
- 水量控制：滑杆或加水按钮记录加入水量。
- 调料控制：滑杆或加调料按钮记录调料量。
- 翻炒操作：按钮或简单锅铲交互记录翻炒次数。
- 出锅操作：玩家自行决定何时出锅。

这些控制不需要做成复杂物理模拟，但必须能产生可见反馈和结构化日志。

## 5. 日志与评价

自由操作的核心价值是成为评价依据。第一版至少记录：

- 火候变化，例如从 30 调到 80。
- 加油量。
- 加水量。
- 加调料量。
- 翻炒次数。
- 出锅时最终火候、油量、水量、调料量。

评价可以根据日志生成批判性反馈，例如：

- 火候偏高，容易糊锅。
- 水加得太多，成品偏稀。
- 调料偏重。
- 翻炒次数不足，受热不均。

第一版不强制把这些参数做成严格评分系统，但必须进入日志，让本地评价和 AI 评价有依据。

## 6. 素材与组件策略

首选使用现成像素素材包：

- Pixel UI pack：面板、按钮、边框、输入框、滑杆。
- Pixel kitchen or restaurant tileset：灶台、台面、墙面、锅、盘子。
- Pixel food icons：番茄、鸡蛋、盐、油、水等。
- Pixel font：如果没有合适中文像素字体，优先保证中文清晰可读。

当前代码中可复用的部分：

- `GameManager` 的页面流转。
- `CookingUI` 的动作入口，但需要重做布局和交互。
- `LogManager` 的日志保存链路，但需要扩展参数记录。
- `ReviewManager` 的本地评价入口，但需要读取新日志参数。
- `MvpSceneBuilder` 的场景生成流程，但需要替换做菜页生成逻辑。

## 7. 暂不实现

以下内容不属于第一版 UI 重构：

- 全页面深度美术重构。
- 复杂物理拖拽厨房。
- 切菜、翻炒、控火小游戏。
- 多锅具、多食材自由组合。
- 排行榜。
- NPC 自动品鉴。
- 图片级 AI 评价。
