# 胡闹厨房项目开发依据

本文档根据现有需求规格说明书、软件设计说明书、计划任务书，以及除用例图外的 PlantUML 设计图整理，用作后续继续开发时的简略依据。

## 1. 项目定位

本项目是一款 2D 做饭模拟游戏，名称为“胡闹厨房 / Cooking Simulator”。玩家在虚拟厨房中完成备菜、做菜、调味、品鉴、保存菜品和查看排行榜等流程。系统核心不是单纯还原烹饪画面，而是把玩家的非线性烹饪操作转化为结构化日志，再结合标准菜谱和 AI 评价生成反馈。

优先目标平台为 PC 端和 Web 端，后续可扩展到 Android、iOS 等移动端。技术方向以 Unity 2022 LTS、C#、JSON、本地持久化和外部 LLM API 为主。

## 2. 核心用户与模式

系统有两类主要玩家角色：

- 厨神：负责选择菜谱、准备食材、做菜、发起品鉴、保存菜品到食单，并拥有“声望”属性。
- 老八：负责浏览厨神食单、选择菜品品鉴、撰写评价，并拥有“生命值”属性。

系统还包含 NPC 老八。NPC 老八不是玩家直接操作的角色，而是在厨神保存新菜品后，由系统读取 Markdown 口味配置并调用 AI 自动生成评价，进而影响厨神声望。

## 3. 游戏主流程

完整流程如下：

1. 用户进入登录界面，输入账号和密码。
2. `UserManager` 校验用户身份，`SaveManager` 读取或创建本地存档。
3. 登录成功后进入模式选择界面。
4. 选择厨神模式时，进入准备阶段；选择老八模式时，进入食单浏览和品鉴流程。
5. 厨神在准备阶段选择菜谱和难度，系统加载准备背景、BGM、菜谱、食材和准备步骤。
6. 厨神完成准备操作后进入做菜阶段，对菜品、调料、锅具执行点击、拖拽、切菜、加热、翻炒、调味、出锅等操作。
7. 系统通过 `StateManager` 校验操作是否合法，更新菜品状态，并通过 `LogManager` 记录每一次有效操作。
8. 厨神可中途或完成后发起 AI 品鉴；品鉴结果可用于继续做菜、重新做菜或保存菜品。
9. 完成菜品后，厨神输入菜名和价格，系统保存图片、日志、评价和食单记录。
10. 保存新菜品后，NPC 老八可自动品鉴；真实老八玩家也可浏览食单并品鉴、评价。
11. 系统根据厨神声望和老八生命值生成排行榜。

## 4. 难度模式

准备和做菜阶段支持两种难度：

- 学习 / 容易模式：显示菜谱、当前步骤、计时器和操作提示，帮助玩家按流程完成烹饪。
- 挑战 / 困难模式：隐藏实时菜谱和提示，玩家依靠记忆完成操作，系统只记录行为并在结束后评分。

无论哪种模式，都必须完整记录有效操作日志。难度差异主要体现在 UI 提示和可见信息上，不应影响底层日志结构。

## 5. 菜谱、日志与评价

菜谱是游戏驱动数据，建议放在 `StreamingAssets/Recipes/*.json`，包含菜名、食材、调料、锅具、标准步骤、动作要求、时间要求和容差。

做菜日志是评价基础，至少记录：

- 用户与菜品 ID
- 阶段：prepare 或 cook
- 动作名、操作对象、时间戳
- 操作参数，例如用量、火力、持续时间
- 操作前后菜品状态
- 最终状态和图片路径

AI 评价时应提交标准菜谱、玩家日志、菜品图片、评价者身份，以及可选 NPC 口味配置。AI 返回内容应包含评分、味道评价、操作评价、外观评价、改进建议、声望变化和生命值变化。AI 调用必须异步执行，失败时不能阻断游戏主流程。

## 6. 核心模块设计

后续实现应优先保持以下模块边界：

- UI 层：登录、模式选择、准备、做菜、品鉴弹窗、保存食单、老八选菜、评价、排行榜。
- 用户模块：`User`、`Chef`、`Laoba`、`NpcLaoba`，管理身份、声望、生命值和 NPC 口味。
- 菜谱模块：`RecipeManager` 解析和校验 JSON 菜谱。
- 做菜引擎：`StateManager` 管理菜品状态机，`LogManager` 记录操作。
- AI 评价模块：`AIReviewService` 负责构建 Prompt、HTTP 调用、结果解析、重试和降级。
- 食单模块：`MenuManager` / `DishManager` 管理菜品保存、浏览和读取。
- 存档模块：`SaveManager` 作为唯一文件访问入口，负责用户、日志、菜品、评价、排行榜等持久化。
- 资源模块：`ResourceManager` 管理背景、BGM、音效和贴图。

UI 层不应直接读写文件；业务逻辑统一通过 `SaveManager` 访问持久化数据。

## 7. 菜品状态机

菜品核心状态建议使用：

`RAW -> CUT -> COOKING -> SEASONED -> DONE`

异常状态包括 `BURNT`。当加热时间超过标准时间的一定倍数时，菜品进入 `BURNT`。从 `BURNT` 可以选择重新做菜，但应产生惩罚，例如降低厨神声望。

## 8. 存储与文件约定

课程项目阶段优先使用本地 JSON 文件和图片文件：

- 菜谱：`StreamingAssets/Recipes/*.json`
- NPC 口味：`StreamingAssets/NPCs/*.md`
- 用户存档：`Saves/Users/{uid}.json`
- 做菜日志：`Saves/Logs/{log_id}.json`
- 菜品图片：`Saves/Images/{dish_id}.png`
- 评价记录：`Saves/Reviews/{rid}.json`

API Key 配置不能提交到版本库。用户密码不能明文保存，应使用哈希或其他安全方式处理。

## 9. 开发优先级

建议先实现最小闭环：

1. 登录与本地存档。
2. 厨神模式、菜谱加载、学习 / 挑战模式切换。
3. 准备阶段和做菜阶段的基础交互。
4. 菜品状态变化与完整操作日志。
5. 出锅、截图、菜品命名定价和保存食单。
6. AI 品鉴接口与失败降级。
7. 老八品鉴、评价、声望 / 生命值变化。
8. NPC 自动品鉴和排行榜。

任何新增功能都应优先检查是否会破坏“菜谱驱动 -> 操作日志 -> AI 评价 -> 食单/排行榜”的核心链路。
