# 最小可玩版本闭环要求

本文档定义“胡闹厨房”第一版最小可玩版本（MVP）的闭环范围。目标不是一次性实现完整系统，而是先做出一个可以登录、选菜、做菜、评价、保存、查看结果的基础版本，验证核心玩法链路是否成立。

## 1. MVP 目标

最小可玩版本必须跑通以下核心链路：

`登录 -> 选择厨神模式 -> 选择菜谱 -> 进入做菜 -> 记录操作 -> 出锅 -> AI 或本地评价 -> 保存菜品 -> 查看食单`

该闭环必须能在本地单机环境完成，不依赖多人联网和真实云端数据库。AI 接口可以接入真实 LLM；若接口不可用，必须有本地降级评价，保证流程不中断。

## 2. 必须包含的功能

### 2.1 登录与本地用户

- 提供简单登录界面，支持输入用户名进入游戏。
- 第一版可暂不实现真实密码校验，但需要为后续 `password_hash` 字段预留数据结构。
- 用户数据保存到本地 JSON。
- 登录成功后进入模式选择界面。

### 2.2 模式选择

- MVP 只要求完整实现“厨神模式”。
- “老八模式”按钮可以显示为未开放或进入占位页面。
- 厨神需要有基础声望属性，初始值可设为 0。

### 2.3 菜谱选择

- 至少提供 1 道可玩的菜谱。
- 菜谱使用 JSON 文件描述，至少包含菜名、食材、调料、步骤、目标动作和时间提示。
- 进入做菜前必须能加载并展示菜谱名称和简要步骤。

### 2.4 做菜交互

第一版做菜界面不追求复杂物理交互，但必须支持一组可点击或拖拽的基础操作：

- 切菜
- 下锅
- 加热
- 加调料
- 翻炒或等待
- 出锅

每次有效操作都要产生操作记录。非法操作可以简单提示“当前不能这样做”。

### 2.5 菜品状态

至少实现以下状态：

`RAW -> CUT -> COOKING -> SEASONED -> DONE`

可以额外实现 `BURNT`，但不是 MVP 必需项。界面需要能展示当前状态，例如文字、颜色变化或替换贴图。

### 2.6 操作日志

做菜过程必须生成结构化日志。每条记录至少包含：

- 动作名称
- 操作对象
- 时间戳或相对时间
- 操作阶段：cook
- 操作前状态
- 操作后状态

出锅后将日志保存到本地 JSON，供评价和后续调试使用。

### 2.7 评价

出锅后进入评价流程。评价分两级实现：

1. 优先调用 AI 评价接口，根据菜谱和操作日志返回评分、评价和建议。
2. 如果 AI 调用失败，使用本地规则生成基础评价，例如根据步骤完成度、顺序错误和缺失动作给出评分。

评价结果至少包含：

- 总分，范围 0-100
- 一段总体评价
- 一条改进建议
- 声望变化值

### 2.8 保存食单

评价完成后，玩家可以输入菜品名称和价格并保存。

保存内容至少包括：

- 菜品 ID
- 用户 ID
- 菜名
- 价格
- 最终评分
- 最终状态
- 日志文件路径
- 评价文本
- 创建时间

保存成功后，玩家可以在食单页面看到刚保存的菜品。

## 3. 暂不纳入 MVP 的功能

以下功能不作为第一版闭环要求：

- 老八完整品鉴流程
- NPC 老八自动品鉴
- 排行榜
- 多菜谱批量导入
- 移动端完整适配
- 复杂动画和音效系统
- 多锅具、多食材自由组合
- 账号注册、密码找回、云端同步
- 图片截图保存和图像级 AI 评价

这些功能可以保留接口或占位 UI，但不应阻塞 MVP 闭环。

## 4. 推荐界面流程

MVP 建议只做以下页面：

1. 登录页：输入用户名，点击进入。
2. 模式选择页：选择厨神模式。
3. 菜谱选择页：展示至少 1 道菜。
4. 做菜页：展示菜品状态、操作按钮、当前步骤提示和出锅按钮。
5. 评价页：展示评分、评价、建议和声望变化。
6. 保存页：输入菜名和价格。
7. 食单页：展示已保存菜品列表。

页面之间必须能顺畅返回或继续，不能出现流程断点。

## 5. 本地文件结构建议

MVP 可使用以下目录：

- `StreamingAssets/Recipes/`：存放菜谱 JSON。
- `Saves/Users/`：存放用户 JSON。
- `Saves/Logs/`：存放做菜日志 JSON。
- `Saves/Dishes/`：存放食单和菜品记录 JSON。
- `Saves/Reviews/`：存放评价 JSON。

所有存档读写统一通过 `SaveManager`。UI 不直接访问文件。

## 6. MVP 验收标准

最小可玩版本完成时，应满足以下标准：

1. 新用户可以输入用户名进入游戏，并生成本地存档。
2. 玩家可以进入厨神模式并选择一份菜谱。
3. 玩家可以完成一轮基础做菜操作并点击出锅。
4. 系统能记录完整操作日志，并保存为 JSON。
5. 系统能生成评价结果；AI 失败时也能继续使用本地评价。
6. 声望能根据评价发生变化并保存。
7. 玩家能保存菜品到食单。
8. 食单页面能显示刚保存的菜品。
9. 关闭后重新进入，用户和食单数据仍然存在。

只要以上 9 项全部满足，即认为 MVP 闭环成立。

## 7. 第一版实现原则

- 先保证流程完整，再优化表现。
- 数据结构必须面向后续扩展，不能写死到 UI 中。
- 做菜操作可以简单，但日志必须可靠。
- AI 是增强体验，不是流程单点故障。
- 每个模块先保留清晰边界，避免后续接入老八、NPC、排行榜时大规模重写。
