# UI 重构待办清单

本文档记录当前下一阶段重点：做菜界面 2D 像素风 HUD 重构。后续继续开发时，应同时阅读 `docs/mvp.md`、`docs/项目开发依据.md` 和 `docs/ui_design.md`。

## 当前基础状态

- 当前开发分支：`feature/mvp-cooking-polish`
- Unity 版本：`2022.3.62f3`
- 当前 MVP 场景：`Assets/Scenes/MVP.unity`
- 默认菜谱：`Assets/StreamingAssets/Recipes/tomato_egg.json`
- 当前基础流程已跑通：登录、模式选择、菜谱选择、做菜、评价、保存菜品、查看食单。
- 当前做菜页已有基础 2D 厨房占位视觉，但主要交互仍是按钮驱动，游戏感不足。

## 当前 UI 重构目标

- 首要目标：把做菜页改造成 2D 像素风厨房 HUD。
- 次要目标：其他页面保持可用和基础统一，不做大规模美术重构。
- 核心交互方向：
  - 准备阶段采用单次交互。
  - 炒菜阶段允许自由调整火候、油量、水量、调料量。
  - 所有关键操作写入日志，作为评价批判依据。

## 待办任务

### 1. 素材包选型与导入

- 选择像素风 UI 素材包，用于面板、按钮、边框、输入框和滑杆。
- 选择像素风厨房或餐厅素材包，用于灶台、锅、台面、盘子和背景。
- 选择像素风食材图标，用于番茄、鸡蛋、盐、油、水等基础元素。
- 确认中文字体策略：优先清晰可读，如无合适中文像素字体，则使用普通中文字体搭配像素 UI。

### 2. 做菜 HUD 布局重构

- 将做菜页从表单式面板改为厨房 HUD。
- 中央区域展示锅、灶台、食材和调料。
- 顶部保留厨神名、声望和当前菜名。
- 底部或侧边放置准备交互、炒菜滑杆和出锅按钮。
- 保证 1280x720 参考分辨率下文本和控件不重叠。

### 3. 准备阶段单次交互

- 实现食材准备交互：点击或拖动食材进入准备区。
- 实现切菜交互：点击案板或食材完成切菜。
- 实现下锅交互：点击或拖动处理后的食材进入锅。
- 每个动作需要更新视觉状态和当前步骤提示。
- 每个有效动作继续写入操作日志。

### 4. 炒菜阶段自由控制

- 增加火候控制滑杆。
- 增加油量控制。
- 增加水量控制。
- 增加调料量控制。
- 增加翻炒操作，记录翻炒次数。
- 玩家自行决定出锅，不使用倒计时自动结束。

### 5. 日志结构扩展

- 扩展操作日志，记录火候变化。
- 记录油、水、调料加入量。
- 记录翻炒次数或每次翻炒操作。
- 记录出锅时最终火候、油量、水量、调料量。
- 保持日志仍由 `LogManager` 统一生成，不让 UI 直接写文件。

### 6. 评价反馈扩展

- 本地评价读取新的炒菜参数日志。
- 增加针对火候、油量、水量、调料量、翻炒次数的文字反馈。
- 第一版不强制复杂评分，只需让评价能批判这些操作。
- AI 评价继续使用已有日志作为输入依据。

### 7. 场景生成与验证

- 更新 `MvpSceneBuilder` 中做菜页生成逻辑。
- 新增 Unity 脚本或素材后，通过 Unity 导入生成 `.meta`。
- 重建 `Assets/Scenes/MVP.unity`。
- 运行 Unity batchmode 验证 C# 编译无错误。

## 暂不做

- 不做全页面深度美术重构。
- 不做复杂物理拖拽厨房。
- 不做切菜、翻炒、控火小游戏。
- 不做多锅具、多食材自由组合。
- 不做排行榜。
- 不做 NPC 自动品鉴。
- 不做图片级 AI 评价。

## 后续开发原则

- 做菜页优先体现游戏感，其他页面优先保持流程稳定。
- 新交互必须写入结构化日志。
- UI 不直接读写文件，继续通过现有服务层保存数据。
- 先完成可玩的 HUD，再追加动画、音效和更复杂的素材表现。
