# 测试文档

## 1. 引言

### 1.1 编写目的

本文档为"胡闹厨房"MVP 阶段的测试计划与测试分析报告，目的如下：

1. 验证 `docs/mvp.md` 中 9 项验收标准全部达成；
2. 验证核心链路「登录 → 模式选择 → 菜谱选择 → 做菜 → 评价 → 保存 → 食单」无阻塞、无逻辑错误；
3. 验证数据持久化可靠性及品鉴降级机制（当 AI 品鉴功能调用失败时采用本地算法进行评价）的健壮性；
4. 记录已知缺陷与限制，为后续迭代提供依据。

### 1.2 背景

| 项目 | 说明 |
|------|------|
| **被测系统** | 胡闹厨房（Cooking Simulator） |
| **技术栈** | Unity 2022.3.62f3c1 + C# |
| **被测场景** | `Assets/Scenes/MVP.unity`（由 MvpSceneBuilder 代码生成） |
| **核心链路** | 登录 → 模式选择（老八/厨神） → 菜谱选择 → 做菜 → 评价 → 保存菜品 → 食单 |
| **测试类型** | MVP 功能确认测试（黑盒为主，关键逻辑由白盒验证） |

**被测模块**：Core 层 `GameManager`（状态机编排）；Models 层 `UserData`、`RecipeData`、`CookingLog`、`DishData`、`ReviewData`；Services 层 `SaveManager`、`RecipeManager`、`LogManager`、`ReviewManager`、`AIReviewService`；UI 层 8 个视图面板；Kitchen 层角色移动与 F 键交互系统。

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| **DishState** | 菜品状态机：`Raw → Cut → Cooking → Seasoned → Done` |
| **DonenessLevel** | 熟度枚举：`Raw → HalfCooked → FullyCooked → Overcooked` |
| **CookingLog** | 做菜操作日志，含 ActionRecord 数组，记录每步操作的 action、target、elapsedSeconds、stateBefore、stateAfter |
| **AI 品鉴** | 调用 OpenAI 兼容 API 进行 AI 评价；失败时降级到基于规则（步骤完成度+熟度）的本地评价 |
| **火候** | FireLevel 0-4，控制烹饪速度倍率（0=关火，4=最大火） |

## 2. 测试策略与范围

### 2.1 测试策略

- **黑盒功能测试为主**：以玩家视角操作完整流程，验证 UI 交互、按钮响应、页面跳转功能能否正常使用；
- **关键逻辑白盒验证**：对 `GameManager.TryApplyAction()` 状态机、`ReviewManager` 评分算法、`SaveManager` JSON 序列化路径进行代码级验证；
- **手工测试**：MVP 阶段不搭建自动化框架，由测试人员手工执行并记录；
- **异常路径覆盖**：刻意触发非法操作、AI 配置缺失，验证品鉴降级与错误提示功能。

### 2.2 测试范围

**包含（MVP 9 项验收标准）**：

1. 新用户登录生成 `Saves/Users/{id}.json` 存档
2. 进入厨神模式并选择菜谱
3. 完成一轮做菜并点击"完成"
4. 记录完整操作日志为 JSON（`Saves/Logs/{id}.json`）
5. 生成本地评价，AI 调用失败时降级
6. 声誉值根据评价变化并持久化
7. 保存菜品到食单
8. 食单页面展示已保存菜品
9. 重启后数据完整恢复

**不包含**：性能测试、安全测试、兼容性测试、老八模式完整流程、排行榜、音频/动画、多锅自由组合。

### 2.3 测试环境

| 项目 | 配置 |
|------|------|
| 操作系统 | Windows 11 Home China 10.0.26200 |
| Unity | 2022.3.62f3c1 Editor Play Mode |

## 3. 测试内容与设计

> 测试项按 MVP 流程排列。格式：标识符 → 前置条件 → 操作步骤 → 预期结果。

### 3.1 登录与用户系统

**T-LOGIN-01 新用户登录**
- 前置：删除 `Saves/Users/` 目录
- 操作：启动 → 输入"测试厨师A" → 登录
- 预期：跳转模式选择页；状态栏显示用户名、声誉=0；`Saves/Users/{id}.json` 含 userId、username、reputation=0

**T-LOGIN-02 老用户回访**
- 前置：T-LOGIN-01 已执行
- 操作：重启 → 输入相同用户名 → 登录
- 预期：加载已有 UserData；声誉值恢复；`lastLoginAt` 更新

**T-LOGIN-03 空用户名校验**
- 操作：不输入文字 → 点击登录
- 预期：提示"请输入用户名"，不跳转

### 3.2 模式选择

**T-MODE-01 厨神模式入口**：点击"厨神模式"→ 跳转自由漫游或菜谱选择页

**T-MODE-02 老八模式占位**：点击"老八模式"→ 显示占位提示，不崩溃

### 3.3 菜谱选择

**T-RECIPE-01 菜谱加载**：进入菜谱选择页 → 显示"番茄炒蛋"，含食材（番茄、鸡蛋）、调料（盐）、步骤序列

**T-RECIPE-02 选择菜谱**：点击"番茄炒蛋"→ 进入 CookingUI，显示菜谱名和当前步骤提示

### 3.4 做菜核心流程（DishState 状态机）

**T-COOK-01 正常完整流程**
- 操作：切菜(Raw→Cut) → 下锅(Cut→Cooking) → 火候调至2档 → 调味(Cooking→Seasoned) → 搅拌 → 完成(Done)
- 预期：每步 DishState 正确更新；CookingLog.records 记录每步 action/target/elapsedSeconds/stateBefore/stateAfter

**T-COOK-02 非法操作拦截**
- 前置：DishState = Raw
- 操作：直接点"下锅"/"调味"/"完成"
- 预期：全部拦截，提示"当前不能这样做"，状态保持 Raw

**T-COOK-03 多食材下锅**：选择番茄下锅(Cut→Cooking) → 选择鸡蛋下锅(Cooking→Cooking)，两个食材独立显示熟度

**T-COOK-04 锅盘互转**：锅中食材→盘子(transfer_to_plate) → 盘中食材→锅(transfer_to_pan)，isInPan 正确切换，状态保持 Cooking

### 3.5 火候与熟度系统

**T-DONENESS-01 火候影响熟度**
- 前置：食材在锅中（鸡蛋 fullCookThreshold=40s）
- 操作：火候=4，观察鸡蛋 Raw→HalfCooked→FullyCooked→Overcooked；火候=0，熟度暂停
- 预期：最大火约 10s 达 FullyCooked，再过约 10s Overcooked；关火后熟度停止增长

**T-DONENESS-02 熟度评分差异**：分别在 FullyCooked/HalfCooked/Raw/Overcooked 完成做菜，FullyCooked 得分最高(+10)，Overcooked 最低(-5)

### 3.6 评价系统

**T-REVIEW-01 本地评价**
- 操作：完成做菜 → 进入评价页
- 预期：显示分数(0-100)、评语摘要、建议、声誉变化值(reputationDelta)；`Saves/Reviews/{id}.json` 已生成且内容一致

**T-REVIEW-02 AI 品鉴降级**
- 前置：删除/错误配置 `ai_review.local.json`
- 操作：完成做菜 → 查看评价
- 预期：正常显示本地评价结果；无崩溃、无卡死

### 3.7 保存菜品与食单

**T-SAVE-01 保存**：输入名称"我的番茄炒蛋"、价格 38 → 保存 → `Saves/Dishes/{id}.json` 生成，跳转至"我的食单"

**T-SAVE-02 丢弃**：点击"丢弃"→ 不生成文件，跳转至"我的食单"

**T-MENU-01 列表展示**：保存≥1菜品 → 进入"我的食单" → 显示名称/价格/评分，按时间倒序，仅显示当前用户菜品

**T-MENU-02 空食单**：新用户进入"我的食单" → 显示空状态，不崩溃

**T-MENU-03 评价详情**：点击菜品 → ReviewUI 展示评分/评语/建议/声誉变化，支持多 NPC 评价切换

### 3.8 数据持久化

**T-PERSIST-01 重启恢复**
- 前置：完成登录→做菜→保存全流程
- 操作：关闭游戏 → 重启 → 同用户名登录 → 进入食单
- 预期：声誉值恢复；菜品列表完整；关联评价可查看；JSON 文件未损坏

**T-PERSIST-02 JSON 结构验证**：检查生成的 UserData（userId/username/reputation/createdAt/lastLoginAt）、CookingLog（records 数组）、DishData（name/price/score/logPath/reviewIds）、ReviewData（score/summary/suggestion/reputationDelta）字段完整且类型正确

### 3.9 NPC 批量评价（后台异步）

**T-NPC-01 后台 AI 批量评价**
- 前置：`ai_review.local.json` 正确配置，`StreamingAssets/NPCs/` 含 npc_list.json
- 操作：保存菜品 → 立即查看食单（不阻塞）→ 约 30s 后重新进入查看菜品详情
- 预期：菜品关联老八/坤坤/挂壁大神/曼巴等 NPC 评价；平均分重新计算；各 Review JSON 已保存

### 3.10 自由漫游与交互

**T-FREEROAM-01 角色移动**：WASD/方向键 → 角色平滑移动

**T-FREEROAM-02 冰箱交互**：靠近冰箱按 F → 显示提示 → 勾选番茄鸡蛋 → 确认 → 携带食材

**T-FREEROAM-03 灶台交互**：携带食材靠近灶台按 F → 进入 CookingUI，食材预选

**整体通过条件**：9 项全部通过，且无致命缺陷。

### 4.2 缺陷等级

| 等级 | 定义 | 示例 |
|------|------|------|
| **致命** | 核心流程阻塞、数据丢失、崩溃 | 登录后无法做菜；菜品重启全丢失；Unity 崩溃 |
| **严重** | 主要功能不可用或结果严重错误 | 评价分始终为0；食单始终为空；火候不工作 |
| **一般** | 功能可用但存在明显偏差 | 熟度显示延迟；步骤提示错误；UI 错位 |
| **轻微** | 不影响功能的瑕疵 | 文字拼写错误；颜色偏差；动画不流畅 |

### 4.3 终止测试条件

1. 出现致命缺陷导致无法继续执行后续用例
2. 核心链路（登录→做菜→保存）任一环节连续 3 次不通过
3. 数据持久化存在系统性缺陷（如所有 JSON 损坏）
4. Unity Editor Play Mode 10 分钟内崩溃超过 2 次

### 5.3 分析摘要
